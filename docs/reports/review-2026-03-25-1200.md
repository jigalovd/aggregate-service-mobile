## ⚠️ MINOR OBSERVATIONS

### 1. Provider-level reschedule restriction (US-3.43)

**User Story:** US-3.43 - Мастер может запретить клиентам переносить записи через настройки профиля.

**Observation:** В текущей реализации `RescheduleBookingUseCase.kt` не проверяется настройка `allow_reschedule` из профиля провайдера.

**Current Implementation:**
```kotlin
// RescheduleBookingUseCase.kt:47-50
val booking = repository.getBookingById(bookingId).getOrElse { error ->
    return Result.failure(error)
}
```

**Analysis:**
- Проверка `allow_reschedule` может быть реализована на backend
- Frontend-only проверка недостаточна (должна быть и на backend)
- Это **НЕ критическое нарушение**, так как backend responsibility

**Recommendation:** Убедиться что backend API `/bookings/{id}/reschedule` проверяет `provider.allow_reschedule` перед выполнением переноса.

**Priority:** MEDIUM (если backend уже проверяет - OK)

---

### 2. Auto-accept mode (US-3.42)

**User Story:** US-3.42 - Мастер хочет включить автоматическое подтверждение всех заявок (auto-accept mode).

**Observation:** Явной проверки `auto_accept` настройки в `CreateBookingUseCase` или `BookingRepository` не обнаружено.

**Analysis:**
- При создании booking через `POST /api/v1/bookings`, backend может автоматически подтверждать заявки если `provider.auto_accept = true`
- Это также backend responsibility
- Frontend создает booking, backend решает PENDING или CONFIRMED

**Recommendation:** Убедиться что backend API правильно обрабатывает auto-accept режим.

**Priority:** LOW (если backend уже обрабатывает - OK)
