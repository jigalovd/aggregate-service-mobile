# Детальные Сценарии Использования (Use Cases)

## Принцип: Service-First Approach
Во всех сценариях пользователь начинает с выбора **ЧТО** он хочет сделать (набор услуг), и только потом выбирает **КТО** (мастер) и **КОГДА** (время).

---

## 1. Этап Выбора Услуг (Service Selection)
**Контекст**: Пользователь находится в категории (например, "Ногти").

**UI**:
*   Экран разделен на вкладки/категории ([как на референсе](/docs/design/reference.png) @/docs/design/reference.png): "Все услуги", "Маникюр", "Педикюр", "Дизайн".
*   Каждая услуга имеет чекбокс (или кнопку "+").
*   Внизу экрана закреплена панель "Корзина" (Floating Action Button / Bottom Sheet).

**Логика**:
1.  Пользователь отмечает чекбоксы:
    *   [x] Снятие гель-лака (20 мин, 200₽)
    *   [x] Маникюр классический (40 мин, 1000₽)
2.  Система динамически обновляет "Корзину":
    *   *Итого*: 2 услуги, **60 мин**, ~1200₽.
3.  В "Корзине" доступны две основные кнопки действия:
    *   **[Выбрать Мастера]** (Path A)
    *   **[Выбрать Время]** (Path B)

---

## 2. Сценарий А: Поиск по Мастеру (Master-First Flow)
*Цель: Пользователь хочет найти лучшего мастера или конкретный салон для выбранных услуг.*

### UC-A1: Фильтрация Мастеров
**Прекондишн**: Выбрано услуг на 60 мин. Нажата кнопка **[Выбрать Мастера]**.

1.  **Поиск**: Система ищет мастеров в радиусе X км, которые:
    *   Оказывают **ВСЕ** выбранные услуги (обе услуги привязаны к мастеру).
    *   Имеют хотя бы один свободный слот >= 60 мин в ближайшие N дней (опционально, для скрытия неактивных).
2.  **Выдача**:
    *   Показывается Карта с пинами мастеров.
    *   Снизу список карточек мастеров (Фото, Рейтинг, Цена за весь набор).

### UC-A2: Выбор Мастера и Фокусировка
1.  **Действие**: Пользователь кликает на карточку мастера "Анна" в списке или на пин на карте.
2.  **Реакция Системы (Важное требование)**:
    *   Карта **автоматически центрируется** на координатах Анны.
    *   Масштаб карты увеличивается (Zoom In), чтобы показать точное расположение входа.
    *   Открывается детальная карточка мастера (Bottom Sheet на пол-экрана).
3.  **Просмотр**: Пользователь видит портфолио Анны и итоговую цену (она может отличаться от базовой, если у Анны свой прайс).

### UC-A3: Бронирование Слота
1.  **Действие**: Пользователь нажимает "Записаться" в карточке Анны.
2.  **Календарь**: Открывается расписание **только этого мастера**.
3.  **Слоты**: Система показывает только те слоты, куда вмещается **60 мин**.
4.  **Финиш**: Выбор времени -> Экран подтверждения.

---

## 3. Сценарий Б: Поиск по Времени (Time-First Flow)
*Цель: Пользователю нужно успеть "сегодня вечером" или "в обеденный перерыв", неважно к кому.*

### UC-B1: Выбор Интервала
**Прекондишн**: Выбрано услуг на 60 мин. Нажата кнопка **[Выбрать Время]**.

1.  **UI**: Открывается календарь/часы.
2.  **Ввод**: Пользователь выбирает:
    *   *Дата*: "Завтра (15 Сент)".
    *   *Время*: "с 18:00 до 21:00" (желаемый диапазон начала).
3.  **Действие**: Нажимает "Показать варианты".

### UC-B2: Поиск Доступных Мастеров
1.  **Алгоритм Поиска**:
    *   Найти мастеров, оказывающих выбранные услуги.
    *   Для каждого мастера проверить расписание на "Завтра".
    *   Найти "окно" размером **60 мин**, которое начинается в интервале **18:00-21:00**.
2.  **Выдача**:
    *   Список мастеров, отсортированный по релевантности (рейтинг, удаленность).
    *   В карточке мастера сразу подсвечивается найденное время: *"Анна: свободна в 18:30"*.

### UC-B3: Быстрое Бронирование
1.  **Действие**: Пользователь нажимает на слот "18:30" в карточке Анны.
2.  **Переход**: Сразу открывается **Экран Подтверждения** (минуя профиль и календарь, так как всё уже выбрано).
    *   *Кто*: Анна
    *   *Что*: Снятие + Маникюр
    *   *Когда*: Завтра, 18:30 - 19:30.

## 4. Сценарий В: Поиск Конкретного Мастера (Direct Search)
*Цель: Пользователь знает имя мастера и хочет записаться именно к нему.*

### UC-C1: Поиск по Имени
1.  **Прекондишн**: Услуги **НЕ** выбраны (пользователь идет от имени).
2.  **Действие**: На Главном экране вводит в поиск "Светлана".
3.  **Результат**: Список мастеров с именем Светлана.
4.  **Выбор**: Кликает на профиль Светланы.

### UC-C2: Выбор Услуг у Мастера
1.  **Контекст**: Пользователь в профиле Светланы.
2.  **Действие**: Видит прайс-лист **именно этого мастера**.
3.  **Выбор**: Отмечает галочками услуги.
4.  **Бронирование**: Жмет "Записаться" -> Выбирает время из графика Светланы.

## 5. Краевые Случаи (Edge Cases)

### EC-1: Конфликт Услуг
*   *Ситуация*: Пользователь выбрал "Стрижка" (Парикмахер) и "Маникюр" (Нейл-мастер).
*   *Проблема*: Один мастер вряд ли делает и то, и другое (в рамках MVP фрилансеров).
*   *Решение*:
    *   **Вариант А (MVP)**: Блокировать выбор услуг из разных корневых категорий. При попытке выбрать вторую показать алерт: *"Нельзя заказать стрижку и маникюр в одной записи. Создайте два разных заказа."*

### EC-2: "Окно" исчезло
*   *Ситуация*: Пользователь выбрал время в Scenario B, но пока думал, слот заняли.
*   *Решение*: На этапе нажатия "Записаться" (перед подтверждением) — повторная проверка доступности API. Если занято — показать: *"Извините, это время только что заняли. Выберите другое."*

### EC-3: Несочетаемые услуги (Exclusive Services)
*   *Ситуация*: В рамках одной категории есть услуги, которые технически нельзя выполнить за один визит (или они взаимоисключающие). Например, "Маникюр без покрытия" и "Покрытие гель-лак" (если логика подразумевает выбор типа обработки).
*   *Решение*: Использовать флаг `is_combinable` в БД.
    *   Если выбрана услуга с `is_combinable = false`, то при попытке выбрать любую другую услугу — сбрасывать предыдущий выбор или показывать предупреждение.
    *   *UI*: Такие услуги могут быть реализованы как Radio Button (один из списка), в то время как обычные — как Checkbox.
